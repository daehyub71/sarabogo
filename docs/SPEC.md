## 1. 목표

**"살아본 사람에게 묻고, 나에게 맞게 떠난다."**

검증된 구조화 후기와 한국관광공사 OpenAPI 데이터로, 은퇴 시니어가 안심하고 지역에서 한 달을 살 수 있도록 돕는 웹 플랫폼.

| 항목 | 내용 |
|------|------|
| 공모전 | 2026 관광데이터 활용 공모전 · 웹·앱 구현 부문 |
| 필수 요건 | 한국관광공사 OpenAPI를 반드시 활용해 구현·고도화 |
| 주 타겟 | 은퇴 시니어 장기체류자 (전 국민 대상, UX·톤은 시니어 친화) |
| 팀 규모 | 2~3인 |
| 심사 마감 | 2026년 10월 (1차·최종 심사) |

### 성공 기준 (10월 심사 시점)

1. MVP 4개 기능이 실사용 가능한 수준으로 배포되어 있다 (Vercel 공개 URL).
2. 후기 시딩 50~80건이 출처 표기와 함께 DB에 적재되어, 빈 화면이 없다. (공공누리 제1유형 공공저작물 추출이 주력)
3. AI 추천 결과가 **DB의 실제 후기 id를 근거로 함께 노출**한다 (환각 방지 증명).
4. 관광공사 운영계정이 승인되어 있다.

## 3. 기능 요구사항

### F1 · 지역 탐색 지도

- **FR-1.1** 랜딩에서 큰 버튼 3종(`바다 근처` / `병원 가까운 곳` / `저예산`)으로 로그인 없이 즉시 진입한다.
- **FR-1.2** 지역 카드는 관광지가 아닌 **생활 지표**를 먼저 노출한다: 병원 접근 시간, 평균 한달실비, 후기 수, 평균 별점.
- **FR-1.3** 지역 상세에서 카카오맵 위에 병원·약국·마트·관광지 마커를 종류별로 구분해 표시한다.
- **FR-1.4** 무장애 관광정보(관광공사)를 시니어 편의 지표로 상세에 노출한다.
- **FR-1.5** 관광공사·심평원 데이터는 야간 배치로 `places` 테이블에 정제 저장하며, **화면 요청 시 외부 API를 직접 호출하지 않는다**(응답 지연·쿼터 방어).

### F2 · 구조화 후기

- **FR-2.1** 후기 입력 항목은 아래로 고정한다. 필수 여부는 출처(`origin`)에 따라 다르다.

  | 항목 | 타입 | 사용자 작성(`user`) | 공공저작물 추출(`public_doc`) |
  |------|------|------|------|
  | 의료접근성 | 별점 1~5 | ✅ | 근거 있을 때만 |
  | 외로움/커뮤니티 | 별점 1~5 | ✅ | 근거 있을 때만 |
  | 교통편의 | 별점 1~5 | ✅ | 근거 있을 때만 |
  | 재방문의사 | 별점 1~5 | ✅ | 근거 있을 때만 |
  | 한달실비 | 숫자(만원) | ✅ | 근거 있을 때만 |
  | 자유서술 | 텍스트 | — | — |
  | 태그 | 문자열 배열 | — | — |

- **FR-2.2** 후기 출처는 `origin` enum(`user` / `public_doc` / `interview` / `editor`)으로 구분한다.
- **FR-2.3** `origin = 'public_doc'` 후기는 **공공누리 제1유형(`KOGL-1`)만 채택**하며, 출처 4종(`source_org`·`source_year`·`source_title`·`source_url`)과 사람 검수(`verified_at`) 없이는 저장할 수 없다 (DB 제약으로 강제).
- **FR-2.4** 공공저작물 추출 시 **원문에 근거가 없는 항목은 `null`로 저장한다.** LLM의 추정 기입을 금지한다.
- **FR-2.5** `null` 항목은 화면에 **"정보 없음"으로 명시 노출**하며, 지역별 평균 산출에서 제외한다 (null을 0으로 취급하지 않는다).
- **FR-2.6** 검수되지 않은 시딩 후기(`verified_at is null`)는 공개하지 않는다 (RLS read 정책으로 강제).
- **FR-2.7** `origin = 'public_doc'` 후기 카드에는 출처표시 배지를 반드시 렌더링한다 — 공공누리 제1유형의 유일한 의무 조건.
- **FR-2.8** 지역별 후기 집계(평균 별점, 평균 한달실비)를 배치로 `regions`에 갱신한다.
- **FR-2.9** 후기 작성은 로그인 필요, 조회는 비로그인 허용.

### F3 · 맞춤 추천 (RAG)

- **FR-3.1** 입력: 예산(만원), 건강 배려 필요 여부, 관심사 태그(다중).
- **FR-3.2** 서버는 조건에 맞는 지역의 구조화 후기를 Supabase에서 쿼리해 LLM 프롬프트에 주입한다.
- **FR-3.3** LLM은 **제공된 후기·관광정보 근거 안에서만** 4주차 코스를 생성하고, 각 항목이 참조한 후기 id를 반환한다.
- **FR-3.4** 응답의 근거 후기 id는 화면에 실제 후기 카드로 렌더링된다 (`courses.based_on_review_ids`).
- **FR-3.5** 후기가 부족한 지역은 "에디터 추천 코스"(선탑재)로 대체해 빈 결과를 만들지 않는다.

### F4 · 공유

- **FR-4.1** 추천 결과는 고유 URL을 가지며, 카카오 공유로 전송 가능하다.
- **FR-4.2** 공유 링크는 비로그인 상태에서 열람 가능하다.
- **FR-4.3** 카카오 공유 미지원 환경에서는 Web Share API로 폴백한다 (PWA 설치 시 네이티브 공유 시트 사용).

### F5 · 관리자 콘솔 (내부 도구)

> **설계 원칙: 관리자 콘솔은 입력 창구가 아니라 정책을 강제하는 게이트다.**
> 손으로 붙여넣는 것은 느린 크롤링일 뿐이다. 저작권법은 복제의 수단(봇/사람)을 구분하지 않는다.
> 따라서 "무엇을 넣느냐"에 따라 **폼 자체가 달라야** 하며, 허용되지 않는 입력은 필드를 만들지 않는다.

- **FR-5.1** 접근은 `profiles.role = 'admin'`만 허용하며, `lib/authz`에서 검사한다 (RLS에만 의존하지 않는다).
- **FR-5.2** 후기 입력 시 **`origin`을 먼저 선택**하면 그에 맞는 폼이 렌더링된다. origin별 허용 범위는 아래로 고정한다.

  | `origin` | 원문 저장 | 필수 입력 | 폼 특성 |
  |---|---|---|---|
  | `public_doc` | ✅ 발췌 가능 | 출처 4종 + 공공누리 유형 | **제1유형 선택 시에만 저장 버튼 활성.** 제3자 저작권 확인 체크박스 필수 |
  | `licensed` | ✅ 가능 | 원문 URL + 동의서 파일 | 동의서 업로드(비공개 버킷) 없이는 저장 불가 |
  | `curated` | ❌ **금지** | 별점·비용 등 사실 + 원문 URL | **원문 붙여넣기 필드를 두지 않는다.** 총평은 직접 타이핑, 400자 상한 |
  | `interview` | — | 대상자 동의 확인 | 팀이 직접 청취한 내용 |

- **FR-5.3** `curated`는 사실(별점·비용·소요시간)만 저장하고 표현을 저장하지 않는다. 저작권은 표현을 보호하지 사실을 보호하지 않는다는 원리에 기댄 경로이므로, **원문 문장을 옮기면 근거가 무너진다.**
- **FR-5.4** `curated` 후기는 `source_domain`을 기록하고, **동일 도메인 편중을 대시보드에 경고**한다. 한 사이트에서 체계적으로 다수를 수집하면 사실만 뽑아도 데이터베이스제작자 권리 침해가 된다.
- **FR-5.5** 시딩 후기는 검수 큐를 거친다. 검수자가 원문과 대조해 승인해야 `verified_at`이 기록되고 공개된다 (FR-2.6).
- **FR-5.6** 관리자 콘솔의 모든 쓰기는 `admin_audit`에 행위자·대상·판단 근거(origin·출처·라이선스)를 남긴다.
- **FR-5.7** 지원 프로그램(A층)과 지오코딩 실패 좌표 보정도 이 콘솔에서 처리한다.
- **FR-5.8** 관리자 콘솔은 시니어 UX 요건(18px·48px·고대비) 적용 대상이 아니다. 내부 도구이므로 밀도를 우선한다.

### 공통 UI · 소셜 피드 패턴

인스타그램/페이스북의 **세로 1열 카드 피드**를 채택하되, 시니어 접근성과 충돌하는 관습은 배제한다.

- **FR-UI.1** 지역·후기 목록은 세로 1열 카드 피드로 표시한다 (사진 + 제목 + 핵심 지표 배지).
- **FR-UI.2** **무한 스크롤을 쓰지 않는다.** 10개 단위로 표시하고 "더 보기" 버튼(48px 이상)으로 이어붙인다.
  → 시니어가 현재 위치를 잃지 않고, 뒤로가기 시 스크롤 위치가 보존된다.
- **FR-UI.3** 하단 탭 바는 **아이콘 + 한글 레이블**을 함께 표기한다. 아이콘만 있는 탭을 만들지 않는다.
- **FR-UI.4** 색·타이포는 `styles/tokens.css` 단일 출처에서만 정의하며, 본문 대비 4.5:1을 만족한다.
  회색 저대비 보조 텍스트(인스타 관습)를 그대로 쓰지 않는다.
- **FR-UI.5** 좋아요·팔로우 등 소셜 압박 요소는 MVP에 넣지 않는다. 카드의 행동은 "자세히 보기"와 "공유" 둘뿐이다.

## 4. 비기능 요구사항

| 구분 | 요구사항 |
|------|----------|
| **시니어 UX** | 본문 최소 18px, 버튼 터치 영역 최소 48×48px, 명도 대비 WCAG AA(4.5:1) 이상 |
| **가입 마찰** | 탐색·조회는 비로그인 허용. 로그인은 후기 작성 시점에만 요구 |
| **성능** | 지역 목록 LCP 2.5초 이내 (외부 API 실시간 호출 금지가 전제) |
| **보안** | 모든 외부 API 키는 서버(Route Handler)에서만 사용. 클라이언트 번들 노출 금지. 예외: `KAKAO_JS_KEY` (도메인 제한 필수) |
| **데이터 보호** | 인가는 **애플리케이션 레이어(`lib/authz`)가 1차 경계**. Supabase RLS는 **심층 방어(2차)** 로 유지한다. DB 어댑터를 교체해도 1차 경계는 살아 있어야 한다 |
| **교체 가능성** | LLM·DB는 **포트+어댑터**로 격리한다. 벤더 SDK는 `adapters/` 안에서만 import하며, 포트 시그니처에 벤더 타입을 노출하지 않는다 |
| **설치형 웹앱(PWA)** | `manifest.json` + 서비스워커로 홈 화면 설치를 지원한다. Lighthouse PWA "설치 가능" 판정을 통과한다 |
| **관측** | 관광공사 API 요청 전량에 앱 식별자 `MobileApp=sarabogo` 부착 (운영계정 승인 심사 대상). `AppName` 파라미터는 존재하지 않으므로 보내지 않는다 |

## 5. 기술 스택 (확정)

| 레이어 | 선택 | 근거 |
|--------|------|------|
| 프론트 + API | Next.js (App Router) / Vercel | 프론트·API 단일 배포 |
| 배포 형태 | **PWA (설치형 웹앱)** | 스토어 심사·수수료 없이 홈 화면 설치. 네이티브 앱은 로드맵 |
| DB·인증·스토리지 | Supabase (PostgreSQL) — **`DbPort` 어댑터의 기본 구현** | RLS·Auth 내장, 인프라 관리 최소화 |
| 배치 수집 | Supabase Edge Functions + Cron | 야간 OpenAPI 수집·정제 |
| AI 추천 | Claude API 또는 OpenAI API — **`LlmPort` 어댑터로 교체** | Route Handler에서 호출, RAG 방식. 공급자는 `LLM_PROVIDER`로 선택 |
| 지도 | 카카오맵 API | 시니어 친숙도 우선 |

> **결정: Cloud Run 제외.** 2~3인 팀에서 인프라를 하나 더 관리하는 건 순손실이며,
> "왜 Cloud Run인가?"라는 심사 질문을 자초할 이유가 없다. 무거운 배치는 Edge Functions로 대체한다.

> **결정: PWA 우선, React Native 병행 금지.** 2~3인이 10월까지 두 벌의 클라이언트를 유지할 수 없다.
> 모바일 퍼스트 레이아웃·터치 타겟·Web Share 폴백을 지금 지켜두면, 이후 네이티브 전환 비용이 크게 줄어든다.

> **결정: LLM·DB를 포트+어댑터로 격리.** LLM은 Q1이 미결이라 어댑터가 필수다.
> DB 어댑터는 당분간 Supabase 하나뿐이지만, 인터페이스를 지금 긋지 않으면 나중에 그을 수 없다.
> 대가는 인가를 애플리케이션 레이어에서 직접 구현해야 한다는 것 — RLS를 유일 경계로 쓸 수 없기 때문이다.

- **C층 · 후기 (신뢰자산)** — 본체는 자체 생성. 콜드스타트는 **공공누리 제1유형 공공저작물 추출**로 해소.

## 8. 제약 조건

### 8.1 반드시 지킬 것

- **C-1** 관광공사 API 요청 전량에 고유 앱 식별자 **`MobileApp=sarabogo`** 를 일관 부착한다. 클라이언트 유틸(`lib/kto.ts`)에서 자동 주입하며, 이를 우회하는 직접 호출을 금지한다.
  > ⚠️ **정정(2026-07-12, 실호출 검증):** TourAPI(KorService2)에 `AppName` 파라미터는 **존재하지 않는다.** 보내면 `resultCode 10 · INVALID_REQUEST_PARAMETER_ERROR(AppName)`으로 요청 전체가 실패한다. 앱 이름은 `MobileApp`으로 전달하며, 운영계정 심사가 요구하는 "일관된 앱 이름"은 이 값으로 충족한다.
  > 또한 KorService2는 **성공은 중첩(`response.header`), 오류는 평평한(`{resultCode,resultMsg}`) 형태**로 응답한다. 두 형태를 모두 검사해야 오류를 성공으로 오인하지 않는다.
- **C-2** 시딩 후기는 **공공누리 제1유형 공공저작물**(저작권법 제24조의2)에서만 확보하며, 출처표시 4종을 함께 저장한다. 제2·4유형과 "제3자 저작권 포함" 부분은 채택하지 않는다.
- **C-2.1** LLM은 원문에서 **추출**만 한다. 원문에 없는 별점·비용을 **생성하지 않는다** (근거 없으면 `null`).
- **C-2.2** 모든 시딩 후기는 사람 검수를 거친 뒤에만 공개된다.
- **C-3** 관광공사 이미지 재배포 시 출처를 표기한다 (상세페이지 + 푸터).
- **C-4** LLM 추천은 반드시 DB의 실제 후기를 근거(context)로 사용한다.
- **C-5** 벤더 SDK(`@supabase/*`, `@anthropic-ai/*`, `openai`)는 `lib/*/adapters/` 안에서만 import한다. lint(`no-restricted-imports`)로 강제한다.
- **C-6** 포트 인터페이스(`DbPort`·`LlmPort`)는 도메인 타입만 주고받는다. 벤더 타입을 시그니처에 노출하지 않는다.
- **C-7** 후기 write 인가는 `lib/authz`에서 검사한다. RLS에만 의존하지 않는다.
- **C-8** **수집 수단이 아니라 행위가 기준이다.** 사람이 손으로 옮겨도 복제는 복제다. 원문을 DB에 저장할 수 있는 근거는 **공공누리 제1유형(`KOGL-1`)** 과 **개별 이용허락(`CONSENT`)** 둘뿐이다.
- **C-9** `curated` 경로는 **사실만** 저장한다. 원문 문장을 옮기지 않으며, 동일 출처 도메인에 편중되지 않게 관리한다.

### 8.2 절대 하지 말 것 (심사 감점 요인)

- ❌ 관광공사 데이터 크롤링 (약관 위반, 운영계정 승인 거부)
- ❌ 구글 리뷰를 DB에 저장 (Places API 정책 위반 — `place_id`만 저장 가능)
- ❌ 유튜브·블로그·구글맵 스크래핑 (약관 위반 + IP 차단 리스크)
- ❌ 출처 표기 없는 관광공사 이미지 재배포
- ❌ API 키의 클라이언트 노출
- ❌ **LLM이 원문 근거 없이 별점·비용을 추정 기입 (합성 후기)** — "검증된 후기"라는 정체성의 자기부정
- ❌ 공공누리 제2·4유형 저작물을 시딩에 사용 (상업 이용·변형 금지 조건 위반)
- ❌ AI Hub 등 재배포 금지 데이터셋을 공개 서비스 DB에 적재
- ❌ **관리자 콘솔에 블로그·커뮤니티 원문을 붙여넣기** — 손으로 하는 크롤링일 뿐이다 (C-8)
- ❌ 동의서 없이 `origin='licensed'` 저장

### 8.3 일정 제약

관광공사 **운영계정 승인에 약 1주가 소요**되며, 개발 이력(로그)·앱이름(`MobileApp`)·실서비스 URL 확인을 거친다.
→ 10월 심사 역산 시 **9월 초에는 신청 가능한 상태**여야 한다.
