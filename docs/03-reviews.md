# 03 · 후기 데이터 전략 (핵심)

> 후기는 우리 서비스의 1차 무기. **크롤링이 아닌, 공공저작물 + 자체 생성·축적**이 원칙.
>
> 콜드스타트는 공공누리 제1유형 공공저작물로 메우고, 그 위에 사용자 후기를 쌓는다.
> 공공저작물은 마중물이지 본체가 아니다.

---

## 왜 크롤링/외부 API가 답이 아닌가

### 구글 지도 리뷰 — 저장 불가
- 구글 Places API는 리뷰를 제공하지만, **place_id를 제외한 콘텐츠는 캐싱·저장 금지**(약관).
- 요청당 리뷰 최대 5개, 지도 사용 시 구글 지도 위 표시 + 로고·출처 필수.
- 근본 문제: 구글 리뷰는 "장소가 좋았나"지 "한 달 살아보니 의료접근성은?"이 아님. 우리 구조화 항목과 불일치.
- **결론: 저장 불가. 단, 장소 상세페이지에서 "구글 평점 ★4.2" 실시간 표시(조회만)는 가능.**

### 유튜브·블로그·구글맵 크롤링 — 비권장
- 각 플랫폼 약관 위반 → API 키 차단·IP 차단 리스크.
- 공모전에서 크롤링 기반은 심사 감점 요인.
- 블로그 후기는 협찬·비정형 → "검증된 후기"라는 정체성과 정면 충돌, 별점화 불가.

---

## 합법적 후기 확보 전략

### ① 공공누리 제1유형 공공저작물 (주력 · 콜드스타트 해소)

지자체가 발간·공개하는 한달살기 결과보고서, 체험수기 모음집, 기자단 결과물 중
**공공누리 제1유형** 마크가 붙은 것.

- 근거: 저작권법 제24조의2(공공저작물의 자유이용) + 공공누리 제1유형.
- 제1유형은 **출처표시만 하면 복제·변형·상업적 이용이 모두 자유**다.
  → 재구성이 의무가 아니라, **원문 발췌를 그대로 DB에 적재해도 합법**이다.
- 출처표시 문구는 기관명·작성연도·저작물명·출처 URL을 모두 포함한다(`reviews`에 컬럼으로 보관).

**반드시 확인할 두 가지 (시딩 체크리스트로 강제)**

| 확인 항목 | 통과 조건 | 불통과 시 |
|-----------|-----------|-----------|
| 공공누리 유형 | **제1유형만 채택** | 제2유형(상업 금지)·제4유형(변형 금지)은 폐기 |
| 제3자 저작권 표시 | "제3자 저작권 포함 부분 제외" 문구가 **본문 수기에 걸리지 않을 것** | 참가자 개인 수기가 여기 해당하면 폐기 |

> 제3자 권리 확인이 실무의 핵심이다. 공공누리 마크가 문서 전체에 붙어 있어도,
> 참가자 개인이 기고한 수기 본문은 저작재산권이 지자체에 없을 수 있다.

### ② LLM 구조화 추출 + 사람 검수 (주력 · 노동량 해소)

①로 확보한 원문을 LLM이 구조화 5항목으로 **추출**하고, 사람이 검수해 확정한다.
사람이 전건을 손으로 별점화하는 방식 대비 처리량이 한 자릿수 배 늘어난다.

**절대 규칙 — 원문에 근거가 없는 항목은 `null`로 둔다.**

- LLM이 "아마 4점쯤"이라고 추정 기입하는 순간 그것은 추출이 아니라 **합성 후기**이며,
  "검증된 후기"라는 서비스 정체성을 스스로 무너뜨린다.
- 별점 5개가 다 찬 예쁜 카드보다, 2개만 차고 나머지가 "정보 없음"인 정직한 카드를 택한다.
- 화면은 `null` 항목을 **"정보 없음"으로 명시 노출**한다. 평균 계산에서도 제외한다.
- 모든 추출 건은 사람 검수(`verified_at`)를 거쳐야 공개된다.

### ③ 참가자 인터뷰 (보완 · 진정성)

지자체 한달살기 실제 참가자 5~10명 인터뷰 후 구조화 입력.
건수는 적지만 "실제 참가자에게 직접 물었다"는 사실이 심사와 신뢰 양쪽에서 무겁다.

### ④ 자체 사용자 후기 (본체 · 시간이 만드는 자산)

서비스 안에서 사용자가 남기는 구조화 후기. ①②③은 이것이 쌓이기 전까지의 마중물이다.

### 검토했으나 채택하지 않은 것

**AI Hub 리뷰 데이터셋** — 이용정책상 비상업·재배포 금지가 기본이고, 상업적 이용은 수행기관과
별도 협의가 필요하다. 공개 서비스 DB에 적재해 배포하는 형태는 승인 없이는 불가하며,
협의 리드타임이 10월 심사 일정과 맞지 않는다.

| 방법 | `origin` | 원문 저장 | 판정 |
|------|------|------|------|
| 구글 리뷰 API | — | ❌ | 참고용 실시간 표시만 |
| 유튜브·블로그 크롤링 | — | ❌ | 금지 |
| **블로그 원문을 사람이 손으로 입력** | — | ❌ | **금지 — 느린 크롤링일 뿐** |
| AI Hub 데이터셋 | — | ❌ | 비채택 (재배포 금지) |
| 공공누리 제1유형 발간물 | `public_doc` | ✅ | **주력** (LLM 추출 + 사람 검수) |
| 원저작자 개별 이용허락 | `licensed` | ✅ | 보완 (동의서 보관 필수) |
| 공개 자료에서 **사실만** 추출 | `curated` | ❌ | 보완 (소량 · 도메인 편중 금지) |
| 참가자 인터뷰 | `interview` | — | 보완 |
| 자체 사용자 후기 | `user` | — | 본체 |

### 수단이 아니라 행위가 기준이다

**"봇이 아니라 사람이 모으면 괜찮다"는 절반만 맞다.**
크롤링 금지에는 두 가지 이유가 겹쳐 있다 — ⑴ 자동 수집이 약관·robots.txt를 위반하고, ⑵ 남의 저작물을 복제해 우리 서비스로 재전송한다.
손으로 옮기면 ⑴만 사라진다. **저작권법은 복제의 수단을 구분하지 않는다.**
잡코리아 대 사람인 사건에서 문제된 것도 "봇을 썼다"가 아니라 "반복적·체계적으로 가져갔다"였다.

원문을 DB에 저장할 수 있는 근거는 **공공누리 제1유형**과 **개별 이용허락**, 이 둘뿐이다.
그 밖의 공개 자료에서는 **사실(별점화 가능한 수치·소요시간)만** 뽑아 우리 문장으로 쓰고, 원문은 URL만 남긴다.
저작권은 표현을 보호하지 사실을 보호하지 않기 때문이다. **원문 문장을 옮기는 순간 이 근거는 무너진다.**

관리자 콘솔은 이 정책을 폼으로 강제한다 (`SPEC.md` F5): `curated` 폼에는 **원문 입력 필드가 없다.**

### 출처 투명성은 약점이 아니라 가점이다

후기 카드에 **"OO시 2025 한달살기 결과보고서에서 발췌 · 공공누리 제1유형"** 배지를 노출한다.
콜드스타트를 숨기지 않고 드러내면, 그것은 데이터 출처 투명성이라는 강점으로 뒤집힌다.

## 구조화 후기 항목 (별점화 대상)

시니어가 궁금해하는 것을 숫자로 받는다. 자유서술은 보조.

| 항목 | 설명 | 타입 |
|------|------|------|
| 의료접근성 | 병원·약국 거리, 응급 대응 | 별점 1~5 |
| 한달실비 | 실제 한 달 총비용(숙박+생활) | 숫자(만원) |
| 외로움/커뮤니티 | 낮을수록 외롭지 않음 = 사람들이 따뜻함 | 별점 1~5 |
| 교통편의 | 대중교통·이동 편의 | 별점 1~5 |
| 재방문의사 | 다시 오고 싶은가 | 별점 1~5 |
| 자유서술 | 한 줄 총평 + 팁 | 텍스트 |
| 태그 | 바다/산/조용함/의료좋음 등 | 배열 |

- **사용자 직접 작성(`origin='user'`)**: 별점 4항목 + 한달실비 필수.
- **공공저작물 추출(`origin='public_doc'`)**: 전 항목 **nullable**. 원문에 근거가 없으면 비운다. 추정 금지.

## "후기 반영" 작동 방식 (RAG)

사용자가 조건(예산·건강·관심사)을 입력하면:

1. Supabase에서 조건에 맞는 지역의 **구조화 후기를 쿼리**로 가져온다. (예: 대전 인근 + 의료접근성 ★4 이상)
2. 그 후기 데이터 + 관광공사 API 관광정보를 **함께 LLM 프롬프트에 주입**한다.
3. LLM은 "이 근거들로 주차별 코스를 짜고, 각 추천의 근거 후기를 함께 제시하라"는 지시를 수행.
4. LLM이 상상으로 짓지 않고 **우리 DB의 실제 후기를 근거(context)로** 쓰므로 환각↓ 신뢰↑.

> 이것이 검색 증강 생성(RAG). "AI가 후기를 반영한다"의 실체.

### 프롬프트 골격 (의사코드)
```
system: 너는 시니어 한달살기 코디네이터다. 반드시 제공된 후기와 관광정보 근거 안에서만 추천하라.
user:
  조건: 예산 90만원, 건강 배려 필요, 관심사 [바다, 조용함]
  후기 근거: [DB에서 가져온 구조화 후기 배열 JSON]
  관광정보 근거: [관광공사 API 결과 JSON]
  → 4주차 코스를 짜고, 각 항목마다 참조한 후기 id를 함께 반환하라. JSON only.
```
